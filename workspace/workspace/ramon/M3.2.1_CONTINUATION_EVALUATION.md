## 3. Root Cause Analysis

### What M3.2.1 Fixed
✅ Added `initOrderSystem()` to load counter from disk (main.cpp:1318)
✅ Changed to `stable_sort` in query_order and refund_ticket

### What M3.2.1 MISSED (Critical Bug)
❌ **Missing `saveOrderCounter()` calls in `cmd_buy_ticket`** (main.cpp:1088-1113)

### Why This Made It Worse

**Before M3.2.1 (no persistence):**
- Counter always starts at 0 on restart
- Collisions exist but are **consistent** across runs
- `std::sort` gives arbitrary but **reproducible** ordering

**After M3.2.1 (half-implemented persistence):**
- Counter loads from disk but **never saves**
- Disk contains stale counter values
- Creates **MORE collisions** and **non-deterministic** ordering
- Progressive degradation across tests as collisions accumulate

### Evidence from Magnus's Analysis

**The Bug Chain:**
1. Test 1-3: Creates orders with timestamps 1-1500, counter = 1500 in memory
2. **Counter NOT saved to disk** (missing `saveOrderCounter()` in buy_ticket)
3. Test 4-5: Program continues, counter loads **stale 0 from disk**
4. New orders get timestamps 1-300, **colliding with existing 1-1500**
5. `stable_sort` with partial ordering → non-deterministic results
6. Result: 90% → 62% degradation

### Confidence Level
**VERY HIGH** - Independently verified by:
- Magnus (data structures architect) - detailed code analysis
- Cora (root cause analyst) - diff analysis
- Iris (requirements analyst) - independent verification
